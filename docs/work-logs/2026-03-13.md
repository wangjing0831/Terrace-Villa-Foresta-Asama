### 3. AWSControlMCPRole セキュリティ改善

#### 問題の発見
MCP サーバーの `AWSControlMCPRole` に過剰な IAM 権限が付与されていた。

#### 実施した対策

**ポリシー整理：**

| ポリシー | 対応 |
|---------|------|
| `AWSControlMCPPolic`（IAM 操作 `Resource: *`） | **削除** |
| `AWSControlMCPExtra` | IAM Resource を `AWSControl*` に限定・PassRole に `scheduler.amazonaws.com` 条件追加 |
| `IAMManagementAccess`（新規） | IAM + CodeBuild を `foresta-asama-*` + `testapp-*` リソースに限定 |

**インフラ対策：**

| 対策 | 内容 |
|------|------|
| IMDSv2 強制 | EC2 メタデータへの v1 アクセスを禁止、ホップ制限 1 に設定 |
| Permission Boundary | `AWSControlMCPBoundary` ポリシーを作成し `AWSControlMCPRole` に適用。IAM ユーザー作成・アクセスキー発行・Organizations 操作等を Deny |

**リスク改善結果：**

| リスク項目 | 改善前 | 改善後 |
|-----------|--------|--------|
| IAM 権限昇格（全ロール対象） | 🔴 高 | 🟢 低 |
| Lambda 任意実行 | 🔴 高 | 🟢 なし |
| CodeBuild 操作範囲 | 🔴 全プロジェクト | 🟢 両プロジェクトのみ |
| SSRF による認証情報盗取 | 🔴 高 | 🟢 低（IMDSv2） |
| 権限の天井なし | 🔴 高 | 🟢 Permission Boundary で制限 |

### 4. Start/Stop testapp バグ修正

**修正：** EC2 Instance Connect（ブラウザターミナル）で SSH 接続し、`sed -i 's/testspp/testapp/g'` で一括修正後 MCP サーバーを再起動。
